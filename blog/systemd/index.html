systemd-networkd

socketcan自动bringup

详情：https://wiki.archlinuxcn.org/wiki/Systemd-networkd

systemd-networkd用来按规则自动配置netdev，正巧can在linux下就是netdev

systemd-networkd虽然可以配置netdev，但不要举一反三去用它配置以太网接口。目前大多数linux发行版都用另外一个叫NetworkManager的软件来管理网络，同时使用两个软件管理网络可能会引起一些奇怪问题。

为什么不用NetworkManager管理CAN接口？因为它没有这个功能。

2025/9/17更新：Ubuntu的最新LTS版本24默认使用netplan作为前端来管理网络了

输入systemctl status systemd-networkd查看systemd-networkd有没有启动：

如果显示inactive则输入systemctl start systemd-networkd启动服务，再输入systemctl enable systemd-networkd让服务开机自启。

在/etc/systemd/network/文件夹下建立规则80-can.network如下：

[Match]
Name=can*

[CAN]
BitRate=1M

此时系统就会自动按规则配置所有CAN接口波特率为1M，并且启动他们，不需要再手动ip link set了。

重启一次系统查看效果，如果不想重启，则重启一次systemd-networkd服务：systemctl restart systemd-networkd

udev

udev可以捕获内核产生的设备添加/移除事件，按配置的规则根据端口号、PID、VID等等信息自动识别并配置设备（赋权、创建符号链接等等）

详情：https://www.jinbuguo.com/systemd/udev.html

固定USB设备+自动赋权

USB设备都有VID(Vendor ID)和PID(Product ID)两个识别符，输入lsusb列出目前插入的所有usb设备：

以Bus 001 Device 003: ID 8087:0029 Intel Corp. AX200 Bluetooth为例：

  • busnum=1

  • devicenum=3

  • vid=8087

  • pid=0029

一般来说只根据vid和pid就可以确定一个唯一的usb设备，但如果有两个完全一样的usb设备，就还需要额外根据busnum和devicenum来确定一个唯一的设备。

假设现在有一个usb转串口的CH340模块吧：

每次使用它之前，都要输入sudo chmod 666 /dev/ttyUSB0来赋权，而且如果有多个一样的模块，就要指定赋权插在某个USB端口上的，PID为多少，VID为多少的设备，仅仅通过/dev下的文件名肯定是区分不开的。

在/etc/udev/rules.d/文件夹下建立规则100-usbserial.rules如下：

KERNEL=="ttyUSB*",
ATTRS{busnum}=="1",
ATTRS{devnum}=="8",
ATTRS{idVendor}=="1a86",
ATTRS{idProduct}=="7523",
MODE:="0666",
GROUP:="dialout",
SYMLINK+="my_usb_serial"

刷新udev：

udevadm control --reload
udevadm trigger

此时在/dev里可以看到一个叫my_usb_serial的文件，它就是经过udev配置之后的，之前看到的ttyUSB0。之后每次插入符合规则的usb设备时，这个文件都会立即出现。

同理也可以利用udev固定USB转CAN模块，像下面这样，就可以在busnum=3, devnum=7的usb转can模块插入之后创建一个叫my_usb_can的网络接口：

ACTION=="add",
SUBSYSTEM=="net",
KERNEL=="can*",
ATTRS{busnum}=="3",
ATTRS{devnum}=="7",
NAME+="my_usb_can",
GROUP:="dialout"

开机自启

把要开机自启的动作编写成一个service，注册给systemd即可实现可控、稳定的开机自启以及fail retry。

比如要让一个ROS节点自启，并且崩溃之后可以自己尝试再自启的话，就先写这样一个启动脚本：

#!/usr/bin/bash

## /home/a/ws/launch.sh

source /opt/ros/humble/setup.bash
source /home/a/ws/install/setup.bash
ros2 launch bringup_pkg bringup.launch.py

然后，编写服务文件control_node.service

  • After=network.target：在系统启动阶段达到network目标之后启动这个服务

  • ExecStart=/home/a/ws/launch.sh：设置entry为启动脚本/home/a/ws/launch.sh

  • Restart=always：自动重启

[Unit]
After=network.target
 
[Service]
Type=oneshot
ExecStart=/home/a/ws/launch.sh
Restart=always


[Install]
WantedBy=multi-user.target

把这个control_node.service复制到/usr/lib/systemd/system目录下，然后执行systemd daemon-reload重启systemd daemon。

此时执行systemctl status control_node就可以看到注册的服务了：

执行systemctl start control_node即可启动服务，执行systemctl enable control_node即可让服务开机自启。

不启动桌面环境

为了节省性能，或者是为了加快开机速度等等，可以修改systemd的默认启动target为multi-user.target，只启动到终端环境里：

systemctl set-default multi-user.target

如果要恢复桌面环境，把默认target改回graphical.target即可：

systemctl set-default graphical.target
