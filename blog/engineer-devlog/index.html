其他一些深度学习相关的玄学方法先不谈，用传统方法做自动机械臂可以按顺序把全流程拆解成下面四个问题：

  1. 给定机械臂末端要伸到哪，

  2. 求达到这个位置每个关节所需的角度，（逆运动学/ik，inverse kinematics）

  3. 规划一条路径让机械臂能从当前位置平滑地移动到这个目标位置，同时规避掉途中的障碍物，（轨迹规划，path planning）

  4. 设计控制器，让机械臂上的电机输出力矩跟踪这个路径。（dynamics/control）

另外如果末端执行器是夹爪或者灵巧手之类的东西，第一步之前就还有一个计算这个合适的末端位置，好让执行器可以顺利把物体夹住的步骤

逆运动学和轨迹规划这两个相对比较困难的问题可以用moveit这个ros框架轻松秒掉，给定机械臂末端的初末位姿或关节的初末位置，它就可以计算出一条关节位置、速度和加速度的时间序列。最后一个方面，也就是控制是主要需要自己做的工作。

不要用ros2的moveit2，截至2025年10月我写这句话时，humble版本的moveit2开发完成度仍然非常低，缺失很多重要功能，完全不可用（例如文档里写了可以通过一个yaml文件配置求解器参数，但是实际上并不行、ros1 moveit的一大堆求解器到了ros2 moveit2里面只有一两个，ui还有bug）

按直觉来说当然可以用pid直接控，但是如果机械臂比较重（压力机械，从一开始就别让他画重！！），PID就可能解决不了问题（调软了抬不动，调硬了抖），解决这个问题的一种效果比较好的方法是建立机械臂的逆动力学模型，基于这个模型叠加一层动力学前馈（和机械重力补偿效果差不多，25赛季的工程验证了一下这个方法，感觉效果还不错，但是没想象中那么好），或者用lqr、mpc这类控制器（26赛季尝试一下）

MPC

MPC本身是一个带约束的优化问题，想要编程实现它就需要用一种具体的求解方法，而且要考虑求解速度，因为控制频率可能是500hz、1000hz，留给计算的时间不多；可以考虑把MPC转化为QP（quadratic programming，二次规划）形式，这样就可以直接利用一些现成的QP求解器（比如qpOASES）来求解。

这个过程中的关键一步就是用具体的离散系统状态空间方程按照MPC的思想把式子列出来，然后变形成QP一般形式。下面这篇文章讲得很详细很明了：

https://zhuanlan.zhihu.com/p/498481729?utm_psn=1863281508229464064

tinympc on stm32

其实tinympc比较好用，速度够快，而且直接把ABQR矩阵扔进去就能算了，还能写python做完仿真之后直接生成c++代码用来部署，真上车的话建议用这个。文档：https://tinympc.org/get-started/examples/

但是这个东西有几个坑，

第一个就是：tinympc是基于系统时不变这一个假设来做性能优化的，这是它能跑在MCU上的很重要的一个前提。因为模型参数不变，所以它可以离线计算一部分东西，最小化要在线计算的内容

在550mhz主频，cache关的STM32H723上，如果扔一个参数不变的，12维状态的四轴无人机系统给他的话，全0初始条件下10000次控制循环的计算时间是3457ms：

但是如果要追求模型的绝对精准，每个控制周期都做一次线性化的话，那tinympc做的优化就白做了，因为系统模型变了，他离线计算的那些参数现在都要在线算，同样初始条件，同样reference，1000次控制循环的计算时间就达到了1510ms：

虽然上面的对比条件很理想化，但是其实已经能看出问题了，如果太完美主义，那就会丢掉tinympc的性能优势，所以实际应用的时候其实不得不做一些变通来保证性能，比如只在参考轨迹更新的时候，把轨迹的末端视为平衡点做一次线性化，其余时候都用上一次线型化得到的系统模型控制

第二个：tinympc生成的c++代码会有很多巨大的全局对象，每个对象里都是超级多的Eigen矩阵（如下图），这些东西放在单片机上很容易搞出内存问题，跳进Hardfault handler，所以它生成出来c++代码之后，先把这些全局对象都改成用new创建在堆区里的对象，经实测这样就不会触发内存问题了

重力补偿/动力学参数辨识

同构缩小的自定义控制器如果要好用的话，就必须要让他有在没有外力的情况下可以悬停的功能，经过实践证明这个功能会给操作手感带来极大的改善。这种功能是通过逆动力学重力补偿来实现的，为了不额外使用pid来补偿模型误差，最好把模型精度搞的稍微高一点，直接从solidworks里查到的动力学参数经实测达不到要求，只能让机械臂有一种轻飘飘的感觉，还达不到完全零重力飘起来的效果

经实测，静态应用下，也就是重力补偿，这个时候不需要知道转动惯量，所以称重+手调质心位置已经很够用了，有一个点需要注意，就是电机转子的质量在称重的时候不好处理，你不可能把电机的转子拆出来称一下多重。对于大机械臂来说，电机转子的质量可能相比于整条link来说只有一点点，所以影响不大。但是在自定义控制器重力补偿这个应用场景下，电机转子可能对于整条link质量来说分量很大，具体怎么消除这个影响，就是凭经验在连杆质量上加一个值来代表电机转子的质量。

比如说对于达妙4310来说，这个值我反复试出来大概是75g左右

如果是要在大机械臂上做动力学前馈，还是最好做一下参数辨识，这样可以把所有物理参数都测出来准确值。

通过参数辨识可以把动力学模型里面的那些m、l、i之类的玩意都测出来很准的精确值，做完之后不仅自定义控制器悬停搞定了，实际装在工程上的那个大机械臂的前馈也会准的多

看一段具体的动力学模型实现（25年工程的，ypp三轴，里面的参数都是从solidworks里直接读）：

function tau = dynamic_model(q_des)

qd_des = zeros(3, 1);
qdd_des = zeros(3, 1);

% q1 = q_des(1);
q2 = q_des(2);
q3 = q_des(3);
dq1 = qd_des(1);
dq2 = qd_des(2);
dq3 = qd_des(3);

l2 = 0.34;
l3 = 0.37;

I1 = 0.2473;
I2 = 0.1246;
I3 = 0.0402;

m2 = 1.84;
m3 = 3.46;

r2 = 0.097;
r3 = 0.150;

g0 = 9.8;

a1 = m2*r2*r2 + m3*l2*l2;
a2 = m3*r3*r3;
a3 = m3*r3*l2;
b1 = (m2*r2 + m3*l2) * g0;
b2 = m3*r3*g0;

M = [I1 + a1*cos(q2)*cos(q2) + a2*cos(q2+q3)*cos(q2+q3) + 2*a3*cos(q2)*cos(q2+q3), 0,                           0;
    0,                                                                            I2 + a1 + a2 + 2*a3*cos(q3), a2 + a3*cos(q3);
    0,                                                                            a2 + a3*cos(q3),             I3 + a2];

M = vpa(M, 4);

N = [-0.5*a1*dq2*sin(2*q2) - 0.5*a2*(dq2+dq3)*sin(2*q2+2*q3) - a3*dq2*sin(2*q2+q3) - a3*dq3*cos(q2)*sin(q2+q3), -0.5*a1*dq1*sin(2*q2) - 0.5*a2*dq1*sin(2*q2 + 2*q3) - a3*dq1*sin(2*q2+q3), -0.5*a1*dq1*sin(2*q2+2*q3) - a3*dq1*cos(q2)*sin(q2+q3);
    0.5*a1*dq1*sin(2*q2) + 0.5*a2*dq1*sin(2*q2 + 2*q3) + a3*dq1*sin(2*q2+q3),                                  -a3*dq3*sin(q3),                                                           -a3*(dq2+dq3)*sin(q3);
    0.5*a1*dq1*sin(2*q2+2*q3) + a3*dq1*cos(q2)*sin(q2+q3),                                                     -a3*dq2*sin(q3),                                                           0];

N = vpa(N, 4);

g = [0;
    b1*cos(q2) + b2*cos(q2+q3);
    b2*cos(q2 + q3)];

g = vpa(g, 4);

tau = M * qdd_des + N * qd_des + g;
tau = double(transpose(tau));

对于自定义控制器来说，由于要做的只是“静止状态下的”“重力”补偿，所以动力学模型其实变成了一个“静力学”模型，那一大串东西可以简化为tau=gtau=gtau=g，而从上面可以看到g向量只和b1、b2两个参数有关，那就只需要让机械臂静止在不同的位置，把关节位置和关节实际输出力矩记录下来做个最小二乘就好了（重力向量方程整理之后可以表示为tau=F(q)cdot[beginarraycb1b2endarray]tau=F(q)cdot [begin{array}{c} b1 b2 end{array}]tau=F(q)cdot[beginarraycb1b2endarray]，b1b2向量叫待辨识参数集，一般记为pipipi，q知道，tau知道，tau等于g，所以b1和b2可以算出来）

下面是做最小二乘用的代码（AI写的）：

function pi_identified = identify_gravity_parameters(experimental_data)
    % experimental_data: N x 6矩阵 [q1, q2, q3, tau1, tau2, tau3]
    
    N = size(experimental_data, 1);
    T = zeros(3*N, 1);
    W = zeros(3*N, 2);
    
    for i = 1:N
        q2 = experimental_data(i, 2);
        q3 = experimental_data(i, 3);
        tau = experimental_data(i, 4:6)';
        
        % 回归矩阵（只包含重力项）
        Y = [0, 0;
             cos(q2), cos(q2+q3);
             0, cos(q2+q3)];
        
        T(3*i-2:3*i) = tau;
        W(3*i-2:3*i, :) = Y;
    end
    
    % 最小二乘估计
    pi_gravity = (W' * W)  (W' * T);
    
    % 提取b1, b2
    b1_identified = pi_gravity(1);
    b2_identified = pi_gravity(2);
    
    pi_identified = [b1_identified; b2_identified];
end

上面的东西做完之后，如果发现对基座倾斜的鲁棒性不好，就要考虑：上面模型里面的重力向量是固定竖直向下的，所以还要在基座上装一个imu，用imu算出来的基座姿态对重力向量做修正，直接用基座系到世界系的旋转矩阵和重力向量相乘，把重力向量转到正确的方向上即可

g0_base = R_base * g0_world

运动学

https://zgzhou.notion.site/PUMA560-a116cd9695d349aa9f338081f013a2a4

运动学分为正解（给关节角度求末端位置）和逆解（给末端位置求关节角度）。这里主要记录一下逆解的思路

逆解算法分为迭代优化求数值解（https://zhuanlan.zhihu.com/p/1889447970438632548）和几何法求解析解两大类，解析解更稳定且开销小，但根据构型不同，机械臂不一定有解析解，关于怎么判断一种6dof串联构型有没有解析解可以遵循两条原则（pieper准则）：

  1. 三个连续的转动关节的轴相交于同一点

  2. 三个连续的转动关节的轴平行

这两年RM比赛流行的六轴臂一般都是这种PUMA560构型（yaw pitch pitch roll pitch roll），具有解析解，且多解情况只有两种，比较简单：

求这种构型的逆运动学的思路是“先定位再定姿”，分析一下可以发现，末端工具的位置只由前三个关节角度决定，用前三个关节把工具放到想要的位置之后，再用后三个关节调整工具的朝向。

第一个大yaw轴的角度很好确定，从上往下看：

第二、第三个pitch关节也很好确定，从侧面看：

臂长l1l1l1、l2l2l2已知，虚拟杆长l3l3l3也已知（j2、end两点间的距离），求两个角度，这是一个解三角形问题，用几何方法算出q2和q3即可

这里涉及到多解问题：elbow-up和elbow-down两种情况，都可以使末端执行器到达相同的位置。特征也比较明显，就是q2是正数还是负数。

前三轴定位，后三轴定姿。末端三关节的轴线交于一点，所以可以把末端的三个关节抽象成安装在这个交点处的roll-pitch-roll万向节。给定末端执行器的姿态，求关节角，本质是分解合成这个旋转的一个ZYZ欧拉角，相关内容在机器人学导论的一开始就有提到，直接去看书就可以，这里不再赘述

把机械臂模型导入moveit

网上的moveit教程好像都是默认已经有了urdf模型，几乎找不到一步步从solidworks到最终导入moveit的教程，所以这里详细介绍一下整个过程

从sw里导出urdf

首先给solidworks装上sw2urdf插件：https://github.com/ros/solidworks_urdf_exporter

在solidworks里面首先做一些处理，这个模型只需要能表示机械臂的外形即可，内部不会影响整个外部轮廓的的传动结构、电机电调等等东西建议都删掉或者压缩掉，这些东西不仅会增加模型复杂度，还可能导致一些意想不到的干涉影响moveit的碰撞检测。限位机构可以保留。

把机械臂摆到你定义的零位，然后每个关节上建一个点、一个轴、一个坐标系，为了后续方便处理，以每个关节的转轴为Z轴：

然后给整个机械臂建一个参考原点，坐标系定义按REP105来（https://www.ros.org/reps/rep-0105.html）（前+X，左+Y，上+Z）：

设好之后，工具->Tools->Export as URDF：

在左边一个关节一个关节的搭建urdf的树形结构，填写link名称、joint名称，坐标系和轴选刚刚第一步建好的对应的即可，Joint Type按实际情况选择，带限位的旋转关节选revolute，可以无限旋转的旋转关节选continuous，平动关节（比如夹爪的指头）选prismatic，最后把这个link包含的零件选上：

搭好之后点Preview and Export（不要点对钩），在弹出的窗口里给每个revolute joint设置限位，由于现在不太好想象每个轴的旋转方向，所以可以先填一个-pi~pi，然后待会导出之后再看着rviz调整，力矩和速度限制随便填一下即可：

写action server

规划完路径之后就需要把轨迹下发给机械臂执行，第一种方案是把轨迹从moveit里面取出来，然后后面想干什么干什么，第二种方案是把机械臂用Action server的方式接入moveit，这种方式更好，因为moveit在执行轨迹时会做一些检查，比如机械臂当前位置和规划的轨迹初位置是否重合之类的

另外有一点很重要，就是MTC并没有让你把规划好的轨迹取出来的接口，如果要执行规划出来的task，就只能走action server，所以建议直接选第二种方案。

具体一点说的话，就是在创建moveit config包的时候，会让你配置controller：

以arm这组的arm_l_controller为例，意思就是：你要自己写一个control_msgs::action::FollowJointTrajectory类型，名字是/arm_l_controller/follow_joint_trajectory的Action Server，用来接收moveit传给机械臂的轨迹指令。接到这个指令之后，依据这个轨迹让机械臂动就好了。

写好之后，moveit就可以接管机械臂了，同理也可以把末端的夹爪组也接进去：

moveit task constructor(MTC)

让机械臂去抓一个东西这个过程其实是一个比较复杂的多阶段任务，举个例子就是要先让机械臂到物体前面、手张开、再直着把手伸过去、闭合夹爪、抬起来一点、送到目标位置前面、稍微伸进去一点、然后把手松开，最后再复位。如果涉及到双臂，那就更麻烦了，会涉及到并行的阶段。

可以想象的到如果只给你一个从哪到哪的轨迹规划api，剩下的东西都手搓的话需要造多少轮子。（怎么把所有阶段联合起来规划最优解、量化代价、处理并行任务之间的互相干涉、阶段之间的对接…）头铁的话就去慢慢搞状态机、行为树、各种逻辑、各种并行编程、对坐标系，还有搞moveit的那一堆稀烂的api吧。至于最后时间允不允许你把这些东西做出来，那只能说祝你好运。

并不是说自己做这些工作不好，这样自己造轮子确实可以带来很多灵活性，工业界+学术界这么干的人很多，而是在RM这种缺人手还要快速出成果的场景下，想把这种方案的完成度做高非常不现实。

搞moveit的那群人意识到这种”task construction”问题的普遍性，所以在moveit上面又造了一层用来解决这种问题的框架，叫mtc。它的功能就是，把一个机械臂任务拆成一些普适性的”stage”，然后用户可以用定义一系列stage的方法来定义一个复杂的任务。stage之间产生的交互，mtc会帮你搞定。

这个东西没有binary，不能apt install，需要从源码编译，具体怎么弄看这篇文章

上下位机的RPC通信

2026/6/9: 用CANFD总线跑RPC是错的

上位机需要给下位机下发轨迹指令，因为我的debug需求所以我还要下位机可以通过给上位机发消息的方式来打log，不管是轨迹还是log字符串都是不定长的，如果你自己搓通信协议那复杂程度不亚于dji搞的那个裁判系统串口协议。除开这个你还要维护双向通信session，还有一堆乱七八糟的，这种东西其实就是ros2的那个service机制，而这种机制其实有一个更一般的名字，叫远程调用（rpc）。

rpc_core

这个技术早已经成熟到不能再成熟了，有grpc、erpc等等老牌框架，但是为了简洁性所以用了一个叫rpc_core的只支持c++的框架，因为它不需要处理跨语言通信所以设计比较简单，而且它几乎自带了所有stl容器的序列化算法，用起来只需要实现一个传输层，然后就可以把各种std:vector、std::string、std::chrono等等发来发去

但是它需要自己实现一个可靠的传输层，它帮你处理数据打包，你自己考虑怎么把打包好的一大串数据可靠的传给对方，这既是优点也是缺点。综合考虑易用性和可靠性我选了canfd当传输层，然后写了一个长数据包分片协议，但是因为can总线有仲裁机制，导致大量丢帧，我不管怎么设计、怎么处理特殊case都没有什么用，连接依然不稳定。

libcanard

https://github.com/OpenCyphal/libcanard

这个项目似乎就能解决问题，libcanard是一个专门给CAN总线设计的rpc框架，后续等我试一试
